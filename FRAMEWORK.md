# P2 Build Framework — how we work, locked before we write code
Governs every P2 session. PLAN.md says WHAT and WHEN; BENCHMARKS.md says HOW MEASURED; this file says HOW WE OPERATE. P1 inherits this framework unchanged when it starts.

## 1. Roles — center of control
- **Shreyas owns:** every research decision (model routes, priors, what to test next), every merge (nothing becomes "the project" until you've read it), every commit and push (your GitHub, your authorship), every claim (a result you can't re-derive on a whiteboard doesn't ship).
- **Claude owns:** code drafts, harness/plumbing, literature pulls, plot generation, writeup drafts, and pushing back when a choice smells like result-chasing.
- **Hard rule:** I never report a number that doesn't exist in a results artifact on disk. If I estimate, I say "estimate" and it never enters a writeup.

## 2. Session protocol (every build session, no exceptions)
1. **Open:** read `STATE.md` (30-second cold start), state the session goal as one sentence tied to the current milestone gate.
2. **Plan:** 5-line plan before code. If mid-session something breaks the plan, stop and re-plan — don't push through (your rule).
3. **Build:** smallest unit that can be verified. One tack per session; parked ideas go to `LOG.md`, not into the code.
4. **Verify before done:** run it. Tests pass, or output reproduced twice with fixed seed. "Would a staff engineer approve this?" — if unsure, it's not done.
5. **Close:** update `STATE.md` (what works now, exact next action), append decisions to `LOG.md`, corrections to `LESSONS.md`. A session may end mid-feature but never mid-state: repo runs at every close.
6. **Extract:** if the session produced a number, plot, or war story → one line into STATE.md's "content hooks" list; the Sunday review picks it up from there.

## 3. Repo & engineering rules (inherited from your own best codebases)
- **Home:** extend `Heston-engine` (per ROADMAP §7 — no new repo sprawl). Layout: `neural/`, `exotics/`, `data/`, `experiments/`, `results/`, `tests/` added beside the existing `models/ calibration/ utils/`.
- **Zero hardcoded constants** — everything in `config/` (your Heston-engine principle).
- **Atomic writes** — `os.replace()` for anything a crash could corrupt (your Orion parquet lesson).
- **Seeds fixed and logged** on every stochastic run; MC path counts printed on every figure.
- **No hand-made numbers:** every table/figure in any writeup is generated by a committed script reading `results/`; regenerating must reproduce it byte-for-byte (modulo timestamps).
- **Tests gate results:** a pricing/calibration result only counts if `pytest` is green at that commit.
- **Windows guards:** `sys.stdout.reconfigure(encoding="utf-8")` in entry points (your cp1252 lesson); paths via `pathlib`.

## 4. Experiment lifecycle — pre-registration, Orion-style
Every question we put to data becomes `experiments/EXP-###-slug/`:
- **`spec.md` written BEFORE any run:** hypothesis (one sentence), metric, pass/fail bar, data split, seed policy, and what we'll conclude in each outcome branch. Once the first run starts, the spec is frozen.
- **`run.py`:** the only way results get produced; writes to `results/EXP-###/`.
- **`verdict.md` after:** pass/fail against the frozen bar, honored regardless. Near-misses are rejects (your PF 1.169 vs 1.30 precedent). Bug-fix vs result-chasing distinction logged explicitly: a re-run is legitimate only if the fix is validated against ground truth independent of the outcome we want.
- **`experiments/REGISTRY.md`:** one line per experiment — id, question, bar, verdict, date. The registry is the project's honesty ledger and, later, the writeup's spine.
- Failed experiments are content: a defended negative is a post (your 0-for-7 graveyard is the precedent and the brand).

## 5. Data governance
- **No Orion desk data in this repo, ever** — even without an NDA, the project's credibility requires clean provenance. Sources: NSE public (bhavcopy, option-chain EOD you collect yourself), your NSE-Analytics pipelines, or documented free sources.
- `data/README.md` records: source, collection date, script, row counts, checksums. Raw data never edited in place; cleaning is scripted, cleaned artifacts are derived files.
- **Time-based splits only.** Test lockbox = final 3 months, untouched until the M2 gate (BENCHMARKS.md). Opening the lockbox early is a logged event requiring a written reason.
- Surface-quality gate before anything downstream: arb-violation rate per snapshot logged; snapshots failing the M0 bar are excluded by rule, not by hand.

## 6. Definition of done (per milestone)
A milestone closes only when: (1) its PLAN.md gate is met — or honestly failed with a verdict and the pre-registered pivot taken; (2) tests green; (3) README section updated; (4) registry current; (5) at least one content hook extracted; (6) you can defend every design choice in it aloud — we run a 10-minute defense drill (why this route, what would break it, what you'd do with 10× compute) and its notes go to `orion/ownership-map.md` §5 story bank.

## 7. State files (the project's memory)
- **`STATE.md`** — current milestone, what works, exact next action, open questions, content hooks. Overwritten each session close.
- **`LOG.md`** — append-only dated decisions with one-line rationale ("chose COS over CF for pathwise gradients because …"). This is where interview depth comes from.
- **`LESSONS.md`** — your intern-skill pattern: every correction becomes a rule; rules get read at session open until they stop being violated.

## 8. Cadence integration
Tue/Thu 3h + Sat 2h = build sessions (say "P2 build session"). The Sunday review reads STATE.md for milestone status and content hooks; posts pull numbers only from `results/`. If two consecutive weeks close without a milestone advancing, the weekly brief flags it and we re-scope in session rather than silently slipping — scope leaves, dates don't.

## 9. Anti-goals (what this framework exists to prevent)
Half-finished twin repos at interview time; a beautiful NN that can't reprice vanillas; results that moved between the run and the writeup; "it works on my machine" states between sessions; quantum-grade claims on classical-grade evidence.
